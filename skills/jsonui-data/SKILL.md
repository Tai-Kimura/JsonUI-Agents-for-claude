## Cross-Platform Data Copy (REQUIRED when source_project_path is provided)

**When `<source_project_path>` is provided, copy the `data` section from the source platform's layout JSON FIRST, then convert types to the target platform.**

### Procedure:
1. Find the corresponding layout JSON in the source project and read its `data` section
2. Copy the `data` section to the target layout JSON
3. **Convert types to the target platform** using the type mapping system

### Type Conversion Reference Files (MUST READ)

The type system auto-converts most types between platforms. To understand valid types and conversions, read these files in the **target platform's** tools directory:

1. **`<tools_directory>/lib/core/type_mapping.json`** тАФ Central type mapping configuration
   - Defines how JSON types map to each platform's native types
   - Handles mode-specific types (e.g., Color тЖТ SwiftUI `Color` vs UIKit `UIColor`)
   - Includes primitive types, collection types, and special types (Color, Image, Visibility)

2. **`<tools_directory>/lib/core/type_converter.rb`** тАФ Type conversion logic
   - `to_swift_type()` / `to_kotlin_type()` / `to_typescript_type()` methods
   - `parse_function_type()` тАФ Converts callback types (e.g., `(() -> Void)?`)
   - `default_value_for()` тАФ Platform-specific default values
   - `Array()` / `Dictionary()` syntax handling

3. **`<tools_directory>/lib/core/binding_validator.rb`** тАФ Callback type validation

### Key Points:
- **Primitive types** (String, Int, Bool, etc.) are auto-converted тАФ use the JSON universal name
- **Collection types** use `Array(ElementType)` / `Dictionary(KeyType, ValueType)` syntax тАФ auto-converted
- **Function types** like `(() -> Void)?` are auto-converted (VoidтЖФUnitтЖФvoid, BoolтЖФBooleanтЖФboolean)
- **Mode-specific types** (Color, Image, Visibility) are auto-converted based on platform and mode
- **Custom/domain types** are NOT auto-converted тАФ keep as-is if they exist on both platforms

**Do NOT redefine data sections from scratch when a source exists. Always copy first, then verify types against `type_mapping.json`.**

## Scanning JSON for Event Handlers

When working on a JSON file, scan for binding patterns and ensure they're in the data section:

### Binding Patterns to Look For

| Attribute | Pattern | Data Type |
|-----------|---------|-----------|
| `onClick` / `onclick` | `"@{onButtonTap}"` | `(() -> Void)` |
| `onValueChange` | `"@{onToggleChanged}"` | `((Bool) -> Void)` |
| `text` (TextField) | `"@{email}"` | `String` |
| `onTextChange` | `"@{onEmailChanged}"` | `((String) -> Void)` |

тЖТ Examples: `examples/binding-missing-data.json` (wrong), `examples/binding-with-data.json` (correct)

## Common Event Handler Types

## 2-Way Binding Default Values (REQUIRED)

For properties bound to text input fields (2-way binding), **ALWAYS specify a `defaultValue`**:

тЖТ Examples: `examples/twoway-binding-correct.json`, `examples/twoway-binding-wrong.json`

**Why this is required:**
- 2-way bindings update the data model when the user types
- Without a default value, the property may be uninitialized
- This can cause crashes or undefined behavior in the generated code

**Common 2-way binding scenarios:**
- `TextField` with `text: "@{inputValue}"`
- `TextEditor` with `text: "@{content}"`
- `SecureField` with `text: "@{password}"`
